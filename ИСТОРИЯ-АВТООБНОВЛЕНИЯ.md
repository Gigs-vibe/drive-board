# Taska PC — как мы сделали авто-обновление (полная история)

> Дата: 30.06.2026. Итог: **авто-обновление работает** (подтверждено end-to-end на связке 1.6.0 → 1.6.1).
> Этот документ — про то, с чем боролись, как пришли к результату и что попутно поменяли.

---

## 1. Чего хотели

Чтобы при выходе новой версии приложение **само** обновлялось — «как в Telegram»:
нажал одну кнопку → новая версия скачалась, установилась поверх старой, приложение
перезапустилось само. Без ручного скачивания exe с GitHub, без кучи старых версий на диске.

## 2. Как это работает в итоге (финальный механизм, v1.6.x)

1. Через 6 секунд после старта `check_update()` в `app.py` берёт последний релиз с GitHub
   (`api.github.com/repos/Gigs-vibe/drive-board/releases/latest`) и сравнивает тег с `VERSION`.
2. Если на GitHub версия новее — показывается **зелёный баннер** в шапке приложения
   (`showUpdateBanner` в `board.html`) и **push-уведомление** Windows (`winotify`).
3. Пользователь жмёт **«Установить»** → вызывается `Api.download_and_update()`:
   - скачивает `TaskaSetup.exe` в `%TEMP%\Taska` (с прогресс-баром);
   - **проверяет, что это реально программа** — первые 2 байта `MZ` и размер > 1 МБ
     (защита от того, что скачался не тот файл);
   - снимает с файла метку «загружен из интернета» (пишет `ZoneId=0` в Zone.Identifier),
     чтобы не мешал SmartScreen;
   - пишет маленький `run_update.bat`: *подождать 3 сек → запустить установщик тихо → удалить себя*;
   - запускает этот bat через **`ShellExecuteW`** (скрытое окно). Важно: процесс, запущенный
     так, **независим от приложения и переживает его закрытие**;
   - вызывает `os._exit(0)` — приложение мгновенно закрывается.
4. `run_update.bat` ждёт 3 секунды (чтобы приложение точно закрылось и отпустило файлы),
   запускает `TaskaSetup.exe /VERYSILENT /NORESTART`.
5. Установщик (Inno Setup) закрывает остатки приложения (`CloseApplications=yes`), ставит новую
   версию поверх в `%LOCALAPPDATA%\Programs\Taska`, и **сам запускает Taska заново**
   (`[Run]` с `Check: WizardSilent`).

Никаких прав администратора не нужно — установка идёт в пользовательскую папку.

## 3. С чем боролись — хронология (и почему каждый шаг не сработал)

### Эпоха 1. Подмена «живого» exe вручную (НЕ заработало)

Сначала приложение было одним файлом `Taska.exe` (PyInstaller `--onefile`). Идея была:
скачать новый exe, подменить запущенный и перезапустить. Windows **не даёт перезаписать
запущенный exe**, поэтому нужен отдельный процесс-«помощник», который переживёт закрытие
приложения. Пробовали 3 способа его отделить:

1. `.bat` через `subprocess.Popen(..., DETACHED_PROCESS)`
2. PowerShell-скрипт через `DETACHED_PROCESS` (+ позже фиксы `Move-Item` и UTF-8 BOM)
3. `.bat` через `cmd /c start "" /min`

**Во всех трёх случаях:** приложение закрывалось и не запускалось заново. На этом этапе
был временный откат на «ручной» вариант (v1.39.1): просто push-уведомление со ссылкой,
скачиваешь exe сам.

### Поворот: сделать exe установочным (Inno Setup)

Ключевая идея, которая всё изменила: **не воевать с подменой живого файла самописными
скриптами, а отдать это установщику**. Установщики (Inno Setup) умеют корректно закрывать
запущенную программу, заменять файлы и перезапускать — это стандартный отработанный механизм.

- Поставили **Inno Setup** (`winget install JRSoftware.InnoSetup`).
- Написали `Taska.iss` — скрипт сборки установщика `TaskaSetup.exe`.
- **v1.4:** приложение теперь ставится как программа (меню Пуск, ярлык), а не портативный exe.

### Проблема А. «Нажал Установить — приложение просто закрылось, и всё»

Перебрали способы запуска установщика так, чтобы он пережил закрытие приложения:
- `ShellExecuteW` (v1.5)
- `subprocess.Popen` с `CREATE_BREAKAWAY_FROM_JOB` (v1.5.1)
- **Task Scheduler** — запуск через `schtasks` (v1.5.4)

Всё равно не ставилось. И что особенно мешало — **ошибки были не видны**: `subprocess.run`
без проверки кода возврата молча проглатывал сбой, и приложение всё равно закрывалось.
Это позже починили — теперь любая ошибка обновления показывается всплывашкой.

### Проблема Б. «Может это SmartScreen / Defender блокирует?»

Проверили по факту (не гадая):
- история угроз Defender по Taska — **пусто**, ничего не блокировал;
- файл качается через Python (`urllib`), у него **нет метки «загружен из интернета»**,
  поэтому SmartScreen его и не трогает.

**Вывод: антивирус и SmartScreen были ни при чём.** Важный урок — проверять фактами, а не подозрениями.

### Проблема В (ГЛАВНАЯ). Скачивался не установщик, а HTML-страница

Вот что на самом деле всё ломало. В коде была **опечатка в одной букве**:
функция `get_asset_url()` искала среди файлов релиза `tasksetup.exe`, а файл называется
`TaskaSetup.exe` → в нижнем регистре это `taskasetup.exe` (с лишней «a»). Имена не совпадали,
функция возвращала «ничего», и код брал запасную ссылку — **на HTML-страницу релиза GitHub**.
В итоге вместо установщика (23 МБ) скачивался HTML (~200 КБ) и запускался как «программа» —
естественно, ничего не происходило.

**Как нашли:** прочитали первые байты скачанного «установщика» — там был `<!DOCTYPE html>`.

**Починили (v1.5.6):** исправили опечатку + добавили проверку `MZ`/размера, чтобы такой
номер больше никогда не прошёл молча.

> Это был корень почти всех провалов «эпохи установщика». Даже когда механизм запуска был
> правильным — он запускал битый файл.

### Проблема Г. Task Scheduler застревал в «Queued»

После фикса опечатки скачивался правильный установщик, но обновление через Task Scheduler
всё равно не применялось: задача вставала в очередь («Queued»), возникала гонка, версия не
менялась. Проверили вручную с логом установки (`/LOG`) — **сам установщик работает идеально**
(«Installation process succeeded»), проблема именно в Task Scheduler.

**Починили (v1.5.8):** выкинули Task Scheduler, вернулись к `ShellExecuteW` + bat с задержкой 3 сек.
После этого **авто-обновление сработало впервые: 1.5.8 → 1.5.9 обновилось само.** 🎉

### Проблема Д. Ошибка «Failed to load python314.dll» при автозапуске

После авто-установки приложение пыталось открыться и падало с
`Failed to load Python DLL python314.dll`. Причина — режим сборки `--onefile`:
такой exe при **каждом** запуске распаковывает себя во временную папку `%TEMP%\_MEIxxxx`,
и в момент автозапуска сразу после установки распаковка конфликтовала (с антивирусом /
остатками старого процесса). Эта же хрупкость раньше всплывала и на другом ПК.

**Починили (v1.6.0):** перешли на сборку `--onedir` — все библиотеки (включая `python314.dll`)
лежат рядом с exe в папке `_internal`, **никакой распаковки в temp**. Запуск стал быстрее и
надёжнее. Пользователь по-прежнему видит только `TaskaSetup.exe` и ярлык — установщик прячет папку.

### Итог

**1.6.0 → 1.6.1 обновилось чисто, end-to-end, без ошибок.** Цель достигнута.

## 4. Что попутно поменяли (не про сам механизм, но сделали по пути)

| Что | Зачем | Версия |
|-----|-------|--------|
| Данные доски переехали в `%APPDATA%\Taska` | Раньше лежали рядом с exe; после установки в `Program Files`-стиле туда писать нельзя, и данные должны переживать обновления | v1.4 |
| Версия в Настройках тянется из приложения (`Api.get_version()`) | Была захардкожена «Версия 1.3», не совпадала с реальной | v1.5.3 |
| Single-instance (один экземпляр) | Раньше при повторном запуске плодились дубли в трее. Теперь через TCP-сокет на порту 27315: повторный запуск находит уже открытое окно и показывает его, а сам выходит | v1.5.3 |
| Видимые ошибки обновления | `download_and_update` теперь возвращает текст ошибки, и он показывается всплывашкой, а не теряется молча | v1.5.4 |
| Проверка `MZ`/размера скачанного файла | Чтобы битый/неправильный файл не запускался как установщик | v1.5.6 |
| Фикс «vv1.5.7» в баннере | В тексте баннера была лишняя буква «v» | v1.5.8 |
| Подчистили залипшую задачу планировщика | Осталась от Task Scheduler-подхода | v1.5.8 |

**Важный фикс из прошлых сессий, который обязательно оставить:**
`pythonnet.load("netfx")` перед `import webview` в начале `app.py`. Без него pywebview
падает на других ПК с ошибкой «.NET runtime not found». `pythonnet` нельзя исключать из
сборки — он обязателен для pywebview на Windows.

## 5. Ключевые файлы

- **`app.py`** — `VERSION`, `check_update()`, `show_update_toast()`, `get_asset_url()`,
  `Api.download_and_update()`, `Api.get_version()`, single-instance (`try_bring_to_front`,
  `start_single_instance_server`).
- **`board.html`** — баннер `#updateBar`, функции `showUpdateBanner` / `updateDownloadProgress`,
  кнопка «Установить» (`updInstallBtn`).
- **`Taska.iss`** — скрипт установщика Inno Setup (onedir, тихая установка, автозапуск).

## 6. Как собрать и выпустить новую версию

```bash
cd "D:/Творчество/dist"

# 1. поднять номер версии в app.py:  VERSION = "1.6.2"

# 2. собрать приложение (onedir!) и установщик
rm -rf build dist/Taska
python -m PyInstaller --noconfirm --windowed --onedir --name "Taska" --icon "icon.ico" \
  --add-data "board.html;." --add-data "icon.png;." --add-data "icon.ico;." app.py
"$LOCALAPPDATA/Programs/Inno Setup 6/ISCC.exe" Taska.iss
# готовый установщик: dist/installer/TaskaSetup.exe

# 3. закоммитить, поставить тег, запушить
git add app.py && git commit -m "v1.6.2: ..." && git tag v1.6.2
git push origin master v1.6.2          # ВАЖНО: пушить КОНКРЕТНЫЙ тег, не --tags!
# (--tags пушит все локальные теги разом и воскрешает удалённые с remote)

# 4. выложить релиз с установщиком (имя ассета ровно TaskaSetup.exe!)
"/c/Program Files/GitHub CLI/gh.exe" release create v1.6.2 \
  "dist/installer/TaskaSetup.exe#TaskaSetup.exe" \
  --repo Gigs-vibe/drive-board --title "Taska v1.6.2" --notes "что нового"
```

> ⚠️ Имя файла в релизе должно быть **ровно `TaskaSetup.exe`** — иначе приложение его не найдёт
> (см. `get_asset_url`). Именно на этом мы один раз обожглись.

Окружение: Inno Setup стоит в `%LOCALAPPDATA%\Programs\Inno Setup 6\ISCC.exe`.
`gh` CLI — в `C:\Program Files\GitHub CLI\gh.exe`. Перед `pip` закрывать Clash/V2Ray (SOCKS мешает).

## 7. Идеи на будущее

- **Подпись кода (code signing).** Сейчас при самой первой ручной установке Windows может
  показать предупреждение SmartScreen (приложение не подписано). На авто-обновление не влияет,
  но для «чистого» вида стоит подписать установщик сертификатом.
- **Прибраться в релизах.** На GitHub много тестовых релизов-пустышек (1.31–1.39, 1.4.01,
  1.5.0x и т.п.) — можно удалить, оставив значимые вехи.
